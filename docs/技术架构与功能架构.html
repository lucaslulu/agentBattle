数字员工矩阵 / MVP 技术架构

先做一台可靠的 Desktop Host，再谈平台化

MVP 目标：2 到 5 个参与方在正式会议前完成一次代理模拟，输出共识、分歧、风险和真人会议议程。先做本地桌面 Host，不先做云端平台。

Desktop-first Mobile-ready Local-first Agent Host on user machine Provider Adapter

一页结论

范围收窄到“桌面主机上的多代理会前对齐”。

推荐技术路线

Tauri 2 + React/TypeScript + Python sidecar + SQLite + 本地文件存储。Tauri 负责桌面壳和本机权限，Python 负责 Simulation Engine、文档抽取和 LLM 接入。

关键约束

手机端先做轻客户端：填写代理卡片、看报告、确认或异议。多代理讨论、文档抽取、导出和日志留在桌面 Host。

为什么不是网页服务

团队没有稳定高性能服务器。把代理编排放在本机，可以少承担在线服务、租户隔离和持续运维压力。

为什么不是 Electron

Tauri 调系统 WebView，包体更轻，本机权限边界清楚。对小团队来说，少一点运行时负担就是少一点维护成本。

为什么要 Adapter

模型、文件抽取、导出方式都可能换。先把接口隔开，后面从 OpenAI 切到本地模型，或者从 DOCX 切到 PDF，不用推倒主流程。

参考：Tauri 官方页。

技术选型

只保留四块：桌面框架、前端、后端、存储。其他能力放进后端模块里处理。

层级 推荐 理由 MVP 做法
桌面框架 Tauri 2 适合 Desktop-first。UI 用系统 WebView，包体比 Electron 轻；本机文件、窗口、权限、打包交给 Tauri。Tauri 2 支持 sidecar，可把 Python 后端作为随应用分发的本地进程。 Windows 先跑通；macOS/Linux 只做打包验证。Rust 只写很薄的一层：启动 Python sidecar、管理进程、处理少量系统权限和前端 IPC。
前端 React + TypeScript 讨论控制台需要复杂状态：阶段、发言、blackboard、证据、干预、报告确认。React/TS 更容易把这些拆成稳定组件，也方便以后复用到移动轻客户端。 Vite 单页应用。页面保留四个主区：会议室、代理卡片、讨论控制台、结果报告。前端只做状态展示和用户操作，不直接拼提示词。
后端 Python sidecar + 薄 Rust bridge 可以用 Python，不影响 Tauri 2。Tauri 仍是桌面框架；Python 作为本地后端进程运行。这样更适合小团队快速写 Simulation Engine、LLM Adapter、文档抽取、报告生成和后续实验。 Python 用 FastAPI/本地 HTTP 或 stdio JSON-RPC 提供接口。MVP 推荐本地 HTTP：前端经 Tauri/Rust bridge 调用，Rust 负责启动、健康检查、退出清理。发布时用 PyInstaller/Nuitka 打包 Python 后端为 sidecar。
存储 SQLite + 本地文件夹 结构化数据进 SQLite：会议室、代理卡片、turn、intervention、blackboard、报告版本。附件、导出文件和模板放本地目录，避免把大文件塞进数据库。 每个 Workspace 一个 SQLite 库；附件按 workspace_id/room_id 分目录；关键对象版本化。后续如需同步，再加 event log，不在 MVP 里做云同步。

MVP 功能架构

功能层不要按“聊天”组织。用户付费买的是会前判断材料，不是几个 AI 互相说话。

创建会议室 标题、目标、背景材料、参与方数量、期望输出。
填写代理卡片 目标、担忧、底线、可让步空间、公开口径、私密备注。
启动代理讨论 立场陈述、互相提问、共识整理、分歧整理、方案生成、真人议程。
人类干预 补充事实、追问、重答、暂停、结束讨论、标记不代表我。
生成报告 共识、分歧、风险、缺失信息、可选方案、议程和行动项。
确认与导出 本方表述确认、异议记录、复制、DOCX/PDF 导出、历史回看。
主持人视角
  • 创建会议室、邀请参与方
  • 启动、暂停、结束代理讨论
  • 查看公开讨论和最终报告
参与方视角
  • 填写自己的代理卡片
  • 补充事实或纠偏代理口径
  • 确认报告中涉及自己的表述
系统代理主持
  • 切换讨论阶段
  • 压缩重复发言
  • 持续更新共识、分歧和待确认点

代理讨论模拟引擎

阶段固定，发言顺序不固定。Manager 根据冲突、证据缺口和人类干预动态调度。

推荐设计：Swarm 内核 + AutoGen Manager 调度

底层采用轻量 Agent 和 handoff；外层 Manager 负责动态选人、追问、收束。再借 AgentX 的 blackboard、schema validation、evidence auditor 和 checkpoint，保证讨论可追踪。

不选固定图编排的原因

会前对齐里，谁该发言取决于上一轮暴露出的分歧。我们保留阶段，不写死发言队列。

Simulation Session

  • 会议目标、参与方、共享材料
  • 阶段、轮次、时间预算
  • 发言、依据、干预日志

Manager Agent

  • 动态选择下一位发言者
  • 决定追问、澄清、总结或推进阶段
  • 触发代理、人类、报告之间的 handoff

Blackboard

  • 共识、分歧、风险
  • 证据、假设、待确认点
  • 报告只从 blackboard 生成
1. 阶段开场 Manager 根据会议目标生成本阶段任务，不指定固定发言队列，只给出本阶段要解决的问题。
2. 候选发言者评分 按冲突相关度、证据缺口、被点名次数、沉默时长和人类干预权重打分。
3. 代理发言或追问 被选中的代理输出结构化发言，也可以 handoff 给更应该回答的人。
4. Manager 更新状态 抽取新共识、新分歧、风险、假设和待确认点；重复内容不进入长期摘要。
5. 继续或收束 如果新增信息不足、预算用完或关键分歧已清楚，就进入下一阶段或生成报告。
模块 职责 MVP 规则 不做什么
Manager 调度下一步动作，维护讨论状态，决定阶段切换。 每次只输出一个 action：speak、ask、clarify、summarize、human_check、advance、stop。 不替参与方做承诺，不把自己的判断写成业务结论。
Speaker Selector 从参与方代理里挑下一位最应该说话的人。 优先选和当前冲突最相关、被追问、长时间没发言、掌握私密线索但可公开表达的人。 不做固定轮询，也不允许同一代理连续刷屏。
Evidence Gate 检查发言是否说明依据。 来源只能是用户输入、共享材料、历史发言、私密备注影响、模型推断。模型推断必须显式标记。 不让代理把猜测写成事实。
Schema Validator 校验 Manager action 和代理发言结构。 缺少立场、依据、待确认点时重答；action 缺字段时不进入下一步。 不接受自由散文式发言直接入库。
Blackboard 沉淀讨论状态。 只保存可复用结论、冲突、证据、假设和待确认点；报告从这里生成。 不把聊天记录当报告骨架。
Memory 保存短期讨论记录和滚动摘要。 完整 turn 入库，LLM 上下文只带压缩后的阶段摘要、关键引用和最近几轮。 不把所有历史一股脑塞进模型上下文。
Checkpoint 保存阶段结果和失败恢复点。 每阶段结束落盘；模型失败、网络失败或人类暂停后可恢复。 不让一次模型失败毁掉整场讨论。
Handoff 把控制权交给另一个代理、人类或报告生成器。 代理可以请求某人回应；Manager 审核后执行 handoff。高风险或承诺类内容交给人确认。 不允许代理私下绕过 Manager 互相无限聊天。
Stop Policy 防止讨论发散和空转。 连续两次无新增信息、达到轮次预算、核心分歧已清楚、用户暂停，任一命中就收束。 不追求 Agent 自己达成完美共识。

发言结构
  • 立场：一句话
  • 理由：最多两条
  • 依据：来源必须可追踪
  • 待确认：需要谁确认什么
阶段结构
  • 开场立场
  • 澄清与追问
  • 冲突映射
  • 方案试探
  • 真人议程
借鉴方案

Swarm 借轻量 Agent 和 handoff；AutoGen 借 Manager 选人；AgentX 借 blackboard、schema validation、evidence auditor 和 checkpoint。

私密备注处理： 私密备注可以影响本方代理的判断，但不能原文进入公共发言。Manager 看到的也应该是可调度信号，例如“该代理有未公开约束，适合询问可公开边界”，而不是直接拿到私密文本。

参考：OpenAI Swarm、AutoGen SelectorGroupChat、AgentX workflow patterns。

Desktop Host 与移动端关系

这部分决定未来怎么走。先把桌面 Host 做可靠，移动端只承担轻任务。

Desktop Host

  • 运行 Python sidecar
  • 管理本地 SQLite 和附件目录
  • 处理文档抽取、报告生成和导出
  • 连接云端或本地 LLM Provider
  • 局域网内提供受控访问入口
局域网 / 同步数据
后续再做

Mobile Light Client

  • 填写和修改代理卡片
  • 查看讨论摘要和最终报告
  • 确认本方表述或提出异议
  • 接收提醒，不负责长耗时编排
  • 不在第一阶段承诺完整离线 Agent Host

功能实现顺序

先把会前对齐闭环跑通，再做漂亮和自动化。每一阶段都要能被试点用户验证。

优先级 功能 为什么先做 完成标准
P0 项目骨架、SQLite 数据模型、会议室 CRUD 没有稳定的本地数据底座，后面的代理讨论都会变成一次性 Demo。 能创建会议室，状态可流转，历史可回看。
P0 代理卡片和共享上下文输入 产品验证的第一件事，是用户愿不愿意输入真实立场。 2 到 5 个参与方能填写卡片；缺字段有提示；私密备注单独存储。
P0 Simulation Engine 最小闭环 产品核心。不能是聊天框，也不能固定轮流发言。 Manager 动态选人；blackboard 可更新；输出带来源；失败可恢复。
P1 讨论控制台 用户要看见议题、立场、共识和分歧在变化。 左侧阶段，中间发言，右侧共识/分歧/干预入口。
P1 人类干预 信任来自控制权。用户必须能打断和纠偏。 补充事实、追问、重答、暂停、标记不代表我会影响后续讨论。
P1 结构化报告 报告才是带进真人会议的交付物。 默认 1 到 2 页，包含共识、分歧、风险、方案、议程和行动项。
P2 确认/异议、依据展开、导出 这部分决定试点用户是否敢把结果带出去。 本方表述有确认状态；结论可展开依据；支持复制、PDF、DOCX。
P2 文档抽取和模板库 真实场景离不开背景材料。 txt、md、docx、pdf 可抽取文本；提供需求评审、销售方案评审、创始人决策模板。
P3 局域网轻客户端、协作工具集成 有试点反馈后再做，否则会稀释桌面 Host 的核心工作。 手机或浏览器能填卡片、看报告、确认/异议；飞书/Slack/Teams 只做发送报告。

Effort 参考

按 2 名全栈/前端 + 1 名偏 Agent/后端 的小团队估算。这里给的是可试点版本，不是商业化平台。

工作包 估算 主要产出 风险
产品信息架构与 UI 骨架 4-6 人日 五个主页面、状态流、基础组件、路由。 讨论控制台很容易滑成聊天界面，需要早做原型走查。
Python 后端与 Tauri bridge 7-10 人日 sidecar 启动、健康检查、本地 HTTP/JSON-RPC、SQLite 接入、错误处理。 Python 打包体积和跨平台路径要提前验证。
数据模型与历史记录 4-6 人日 MeetingRoom、AgentCard、DiscussionTurn、Intervention、Report。 代理卡片和干预记录要版本化，否则后面解释不清。
Simulation Engine 10-14 人日 Manager、handoff、blackboard、schema validation、证据审计、checkpoint。 难点在动态调度和证据约束，不在单次提示词。
LLM Provider Adapter 3-5 人日 OpenAI-compatible 接口、模型配置、用量/错误记录。 本地模型效果参差，MVP 先当可选项。
文档抽取与导出 5-7 人日 txt/md/docx/pdf 抽取、报告 HTML 预览、PDF/DOCX 导出。 PDF 抽取质量不稳定，失败提示要讲人话。
试点场景和 QA 6-8 人日 三条真实场景、隐私边界测试、5 代理 6 轮压测、可用性修正。 如果只测“模型聪不聪明”，会错过真正的产品风险。
总量判断： 4 到 6 周可做试点版。先砍掉云端 SaaS、完整移动端 Agent、长期组织记忆、复杂权限和完整知识库。
